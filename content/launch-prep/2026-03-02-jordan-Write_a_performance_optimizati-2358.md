## Performance Optimization Checklist for Express.js + PostgreSQL App on Railway

This checklist covers key areas for optimizing your Express.js application running on Railway, focusing on database performance, query optimization, caching, CDN, and compression.  Each section provides guidance and considerations.

**I. Database Optimization (PostgreSQL)**

* **[ ] Database Indexing:**
    * **Identify Slow Queries:** Use database monitoring tools (like pgAdmin's query analyzer, Railway's built-in metrics, or New Relic) to pinpoint queries with slow execution times.
    * **Analyze Query Plans:**  Review the query execution plans generated by PostgreSQL to understand how data is being accessed.
    * **Add Indexes Strategically:**  Index columns frequently used in `WHERE`, `JOIN`, `ORDER BY`, and `GROUP BY` clauses.
        * **Consider Index Types:**  Evaluate different index types (B-tree, Hash, GiST, GIN) based on data types and usage patterns. B-tree is usually the best starting point.
        * **Composite Indexes:**  For queries frequently using multiple columns in `WHERE` clauses, consider composite indexes.
        * **Avoid Over-Indexing:** Too many indexes can slow down write operations. Regularly review and remove unused indexes.
    * **Analyze and Vacuum:** Regularly run `ANALYZE` (updates statistics for the query planner) and `VACUUM` (reclaims space from deleted rows) to maintain optimal performance.
* **[ ] Query Optimization:**
    * **Avoid `SELECT *`:**  Only select the columns you actually need. Reduces data transfer and memory usage.
    * **Use `JOIN` Wisely:** Ensure `JOIN` columns are indexed.  Consider the `JOIN` type (INNER, LEFT, RIGHT) and its impact on performance.
    * **Rewrite Subqueries:**  Often, subqueries can be rewritten as `JOIN`s for better performance.
    * **Use `LIMIT` and `OFFSET` Carefully:**  For pagination, use `LIMIT` and `OFFSET` strategically. For large offsets, consider keyset pagination (using a `WHERE` clause based on a unique identifier) instead of `OFFSET`.
    * **Optimize `WHERE` Clauses:**  Use the most efficient operators. Avoid using functions in `WHERE` clauses that prevent index usage.
    * **Use `EXPLAIN`:**  Use the `EXPLAIN` command in PostgreSQL to understand query execution plans and identify potential bottlenecks.
* **[ ] Database Configuration:**
    * **Connection Pooling:**  Railway automatically manages connection pooling. Ensure the pool size is appropriate for your application's load. (Review Railway’s connection pool settings)
    * **Memory Allocation:**  Review PostgreSQL’s memory settings (shared_buffers, work_mem) – consult the PostgreSQL documentation for recommended settings based on your workload.
    * **Autovacuum Configuration:** Fine-tune autovacuum settings (autovacuum_vacuum_scale_factor, autovacuum_analyze_scale_factor) for optimal performance and data consistency.

**II. Application Layer (Express.js)**

* **[ ] Efficient Routing:**
    * **Minimize Route Handlers:**  Keep route handlers concise and focused.
    * **Route Parameters:**  Use route parameters efficiently.
    * **Avoid Excessive Middleware:**  Review and remove unused middleware.
* **[ ] Code Optimization:**
    * **Profiling:** Use a profiler (e.g., Node.js inspector) to identify performance bottlenecks in your code.
    * **Asynchronous Operations:**  Utilize `async/await` or Promises to handle asynchronous operations efficiently.
    * **Avoid Blocking Operations:** Don't perform CPU-intensive tasks directly in the request handler. Offload them to background processes or message queues.
