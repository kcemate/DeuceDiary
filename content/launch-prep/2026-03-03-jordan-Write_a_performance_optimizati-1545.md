## Performance Optimization Checklist: Express + PostgreSQL

This checklist outlines key areas to consider when optimizing the performance of an Express.js application connected to a PostgreSQL database. It's broken down into categories with varying levels of complexity.

**I. Database Optimization (PostgreSQL)**

* **[ ] Indexing:**
    * **Analyze Queries:** Use `EXPLAIN ANALYZE` to identify slow queries and pinpoint missing indexes.
    * **Index Relevant Columns:** Create indexes on columns frequently used in `WHERE`, `JOIN`, and `ORDER BY` clauses.
    * **Composite Indexes:** Consider composite indexes for queries that filter on multiple columns.
    * **Index Types:** Evaluate different index types (B-tree, GiST, GIN, BRIN) based on your data types and query patterns.  B-tree is usually a good starting point.
    * **Regularly Review Indexes:** Don't just create indexes; remove unused ones to avoid slowing down writes.
* **[ ] Query Optimization:**
    * **Use `EXPLAIN`:**  Understand the query execution plan. Look for full table scans and inefficient joins.
    * **Avoid `SELECT *`:** Only select the columns you actually need. Reduces data transfer and processing time.
    * **Use `JOIN`s Efficiently:** Understand different `JOIN` types (INNER, LEFT, RIGHT, FULL) and choose the appropriate one.
    * **Optimize `WHERE` Clauses:** Use indexed columns and avoid complex expressions in `WHERE` clauses.
    * **Subqueries vs. Joins:**  Sometimes, rewriting a subquery as a join can improve performance.  Test both approaches.
    * **Prepared Statements:** Utilize prepared statements to avoid repeated parsing and compilation of the same query.
    * **Limit Results:** Use `LIMIT` to restrict the number of rows returned, especially when pagination is involved.
* **[ ] Database Configuration:**
    * **`shared_buffers`:** Allocate sufficient memory to PostgreSQL's shared buffer cache.
    * **`work_mem`:** Allocate memory for sort operations and other internal processes.
    * **`maintenance_work_mem`:** Memory used for maintenance operations (e.g., `VACUUM`, `ANALYZE`).
    * **`effective_cache_size`:**  Inform PostgreSQL about the amount of memory available to the operating system for caching.
    * **`wal_buffers`:** Memory used for write-ahead logging.
    * **`max_connections`:** Set a reasonable limit to prevent excessive connections.
* **[ ] Database Maintenance:**
    * **`VACUUM`:** Regularly run `VACUUM` to reclaim space and update statistics.  Consider `VACUUM FULL` cautiously.
    * **`ANALYZE`:** Run `ANALYZE` to update table statistics, which helps the query optimizer.
    * **`pg_stats`:**  Monitor the statistics generated by `ANALYZE` and manually adjust them if necessary.


**II. Express.js Application Optimization**

* **[ ] Connection Pooling:**
    * **Use a Connection Pool:**  Never create a new database connection for each request. Use a connection pool (e.g., `pg-pool`) to reuse connections efficiently.
* **[ ] Route Optimization:**
    * **Efficient Route Definitions:**  Minimize the number of routes.
    * **Route Grouping:** Utilize route grouping (Express 4.16+) to simplify route definitions and attach middleware consistently.
* **[ ] Middleware Optimization:**
    * **Minimize Middleware:**  Only include middleware that is absolutely necessary.
    * **Order Matters:** Place performance-critical middleware (e.g., authentication) earlier in the stack.
